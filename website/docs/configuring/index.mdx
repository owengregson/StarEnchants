# Configuring content

Everything StarEnchants reads lives under `plugins/StarEnchants/` as plain YAML. Edit a file and run
`/se reload`, or manage it all from the in-game GUIs. This section walks through the whole layout and
then takes each content type — enchants, sets, crystals, items — field by field.

## The data folder

```text
config.yml     global toggles · combat caps · souls · crystals · heroic · lore · integrations · reload
content/       the content library
  ├─ tiers.yml      rarity tiers (color · weight · glint)
  ├─ enchants/      one file per enchant
  ├─ crystals/      one file per socketable crystal
  └─ sets/          one file per armor set
items/         one file per physical item — books · scrolls · dust · soul gem · orb · nametag · …
menus/         in-game GUI layout overrides
packs/         saved config-pack snapshots (/se pack)
lang.yml       every player-facing message & GUI string
```

:::tip Keys are path-derived
An enchant at `content/enchants/lifesteal.yml` has the key `lifesteal`. Organize files into
subfolders freely — the key is just the path under `content/`. The file path *is* the identity, so
moving or renaming a file changes its key.
:::

## config.yml — the cross-cutting knobs

`config.yml` holds the settings that aren't a single item's look and aren't player-facing text: the
master feature switches, combat caps, and tuning for souls/crystals/heroic/slots/lore/integrations.
Most values are re-read live on the next use, so a `/se reload` re-tunes them without a restart (the
exceptions — integration discovery, the auto-reload interval, the command trigger, and the
slots/souls/scrolls *listeners* — are armed once at boot and noted below).

```yaml
features:                     # master on/off switches for whole subsystems
  enchants: true              # enchant effects fire
  sets: true                  # armor-set bonuses apply
  crystals: true              # crystal effects fire
  heroic: true                # heroic flat stats feed the heroic damage stage
  slots: true                 # the slot orb can be applied (listener armed at boot)
  souls: true                 # the soul gem system (listener armed at boot)
  scrolls: true               # scroll-family interactions (listener armed at boot)

combat:
  max-bonus-damage: -1.0      # ceiling on summed outgoing-damage fraction; <0 = uncapped
  max-bonus-reduction: -1.0   # ceiling on summed damage-reduction fraction; <0 = uncapped
  pvp: true                   # apply combat effects in player-vs-player hits
  pve: true                   # apply combat effects in player-vs-environment hits

messages:
  prefix: ""                  # e.g. "&8[&dSE&8] " — prepended to plugin chat messages
  feedback: true              # send keyed gameplay-feedback messages to players

slots:
  base: 9                     # enchant slots on a fresh item (the hard cap lives in items/slot-orb.yml)

souls:
  deposit-on-any-kill: true   # banks souls into a carried gem on ANY kill

crystals:
  slots: 1                    # crystal sockets per item (separate from enchant slots)
  max-stack: 16               # absolute max crystals one item may ever hold

heroic:
  max-outgoing-factor: 4.0    # ceiling on the heroic outgoing-damage multiplier (>= 1.0)

lore:
  enchant-color: "&7"         # an enchant's display name
  level-color: "&f"           # the level numeral
  crystal-color: "&b"         # a crystal line
  roman: true                 # levels as Roman numerals (Venom III) vs Arabic (Venom 3)
  unknown-label: "&8Unknown Enchant"  # shown for a stored key no longer in the catalog

integrations:
  protection: true            # discover region/claim plugins for the build gate (boot-time)
  economy: true               # discover an economy backend for money effects (boot-time)
  named: {}                   # per-integration off switch, e.g. named: {worldguard: false}

reload:
  re-resolve-players: true    # re-apply worn state to online players after a content swap
  auto-seconds: 0             # >0 = periodic auto-reload interval in seconds (armed at boot)

command-trigger:
  enabled: true               # register a player /cast command for COMMAND-trigger enchants (boot-time)
  name: cast                  # the command players run (no slash): /cast
  description: "Trigger your COMMAND enchantments."
```

:::warning Boot-time vs. live
`integrations`, `reload.auto-seconds`, `command-trigger`, and the `slots`/`souls`/`scrolls` feature
switches are armed **once at server start**. Changing those takes effect on the next restart, not on
`/se reload`. Everything else re-reads live.
:::

## An annotated enchant

One enchant is one file under `content/enchants/`. Here is a real example —
`content/enchants/frostbite.yml` — annotated field by field:

```yaml
tier: uncommon                # rarity → lore color / glint / sort weight (see tiers.yml)
display: "&bFrostbite"        # the name shown on gear lore ('&' color codes)
description: "Chill and slow the enemy you strike."   # a string, or a list of lore lines
trigger: ATTACK               # the event that fires it — one trigger, or a list [ATTACK, BOW]
applies-to: [SWORD, AXE]      # which item kinds can carry this enchant
group: combat                 # shared cooldown / suppression bucket
levels:
  1:
    chance: 25                # percentage roll — fires ~1 swing in 4
    effects:                  # the actions that run when it fires
      - { POTION: { effect: SLOWNESS, level: 1, duration: 60, who: "@Victim" } }
      - { PARTICLE: { particle: CLOUD, count: 6 } }
  2:
    chance: 35
    effects:
      - { POTION: { effect: SLOWNESS, level: 2, duration: 80, who: "@Victim" } }
      - { POTION: { effect: WEAKNESS, level: 1, duration: 60, who: "@Victim" } }
  3:
    chance: 45
    cooldown: 0               # ticks between firings (20 ticks = 1 second)
    effects:
      - { POTION: { effect: SLOWNESS, level: 2, duration: 100, who: "@Victim" } }
      - { POTION: { effect: WEAKNESS, level: 1, duration: 80, who: "@Victim" } }
      - { MESSAGE: { text: "&bFrost grips your foe!" } }
```

### Root keys

| Key | Meaning |
| --- | --- |
| `display` | the enchant's name in lore (`&` color codes) |
| `description` | flavor text — a string or a list of lines |
| `tier` | rarity id from [`tiers.yml`](./index.mdx#rarity-tiers--tiersyml) |
| `trigger` | the firing event — one [trigger](/docs/reference/triggers), or a list |
| `applies-to` | item kinds that can carry it (`SWORD`, `AXE`, `HELMET`, `PICKAXE`, `BOW`, `WEAPON`, …) |
| `group` | shared cooldown / [suppression](./index.mdx) bucket |
| `repeat` | period in ticks for a `REPEATING` enchant (e.g. `100` = pulse every 5s) |
| `requires` / `blacklist` / `removes-required` | apply-relationship rules between enchants |

### Per-level keys

Inside each `levels:` entry:

| Key | Meaning |
| --- | --- |
| `chance` | percentage activation roll (0–100) |
| `cooldown` | ticks before it can fire again (20 ticks = 1 second) |
| `soul-cost` | souls spent per activation (requires the player be in soul mode) |
| `condition` | a [boolean test](/docs/reference/conditions) that gates the firing |
| `effects` | the list of [effects](/docs/reference/effects) to run |

:::tip Set a key once at the root
Any per-level key may instead be set **once at the root** as the default for every level. If every
level shares a cooldown, write `cooldown: 100` at the top level instead of repeating it.
:::

### Writing effects

Effects accept parameters two equivalent ways:

```yaml
effects:
  - "IGNITE:60"                                    # terse — positional args
  - { IGNITE: { duration: 60 } }                   # verbose — named args
  - { POTION: { effect: REGENERATION, level: 1, duration: 60, who: "@Self" } }
```

Two parameter names are special in the verbose form:

- **`who:`** chooses the [selector](/docs/reference/selectors) — *who* the effect lands on
  (`@Self`, `@Victim`, `@Attacker`, `@Aoe{r=3}`, …). Every effect has a sensible default target if
  you omit it.
- **`wait:`** inserts a delay, in ticks, before that single effect runs.

### Conditions

A `condition:` is a boolean expression over `%scope.name%` variables, combined with `&& || ! ( )`
and the operators `== != < <= > >=`, `contains`, and `matchesregex`:

```yaml
condition: "%victim.health% < 6 && %onground%"
condition: "%victim.helditem% contains \"SWORD\""
```

A condition may also end in a **flow clause** — `: %stop%`, `: %force%`, `: %allow%`, or
`: +15 %chance%` (nudge the activation roll). Without a clause it simply gates whether the enchant
fires. See the full [conditions reference](/docs/reference/conditions).

### A richer level

A gated, cooled-down crit with a targeted effect, particle, and message (from
`content/enchants/executioner.yml`):

```yaml
  3:
    chance: 50
    cooldown: 0
    condition: "%victim.health% < 6"
    effects:
      - { DAMAGE_MOD: { side: attack, mode: flat, amount: 8 } }
      - { PARTICLE: { particle: CRIT, count: 18 } }
      - { MESSAGE: { text: "&4&lExecuted!" } }
```

## Rarity tiers — tiers.yml

Tiers are defined once, in `content/tiers.yml`:

```yaml
default-tier: common
tiers:
  common:    { color: "&7",   weight: 10, glint: false }
  uncommon:  { color: "&a",   weight: 20, glint: false }
  rare:      { color: "&b",   weight: 30, glint: false }
  epic:      { color: "&d",   weight: 40, glint: true  }
  legendary: { color: "&6",   weight: 50, glint: true  }
  mythic:    { color: "&c&l", weight: 60, glint: true  }
```

Each tier id carries a lore `color`, a GUI sort `weight`, and whether items of that tier `glint`.
`default-tier` is used by any enchant that omits `tier:`. See
[Rarity tiers](../concepts/rarity-tiers.mdx) for the concept.

## Defining a set

A set lives in `content/sets/`. It declares its pieces, a `complete` threshold, the bonus that fires
once the full set is worn, and an optional matched weapon. From `content/sets/inferno.yml`:

```yaml
display: "&cInferno"
description: "Burn attackers and shrug off fire when the full set is worn."
complete: 4                   # pieces needed for the bonus to activate
armor:
  lore:                       # extra lore stamped on each piece
    - "&cInferno Set &7(4 pieces)"
    - "&7Wear the full set to burn attackers."
  pieces:
    helmet:     { material: DIAMOND_HELMET,     name: "&cInferno Helm" }
    chestplate: { material: DIAMOND_CHESTPLATE, name: "&cInferno Chestplate" }
    leggings:   { material: DIAMOND_LEGGINGS,   name: "&cInferno Leggings" }
    boots:      { material: DIAMOND_BOOTS,      name: "&cInferno Boots" }
  trigger: DEFENSE            # the set bonus is a trigger + effects block, like an enchant
  chance: 45
  cooldown: 100
  effects:
    - { IGNITE: { duration: 80, who: "@Attacker" } }
    - { POTION: { effect: FIRE_RESISTANCE, level: 1, duration: 100, who: "@Self" } }
    - { PARTICLE: { particle: FLAME, count: 12 } }
weapon:                       # optional — fires only while the full set is worn
  material: DIAMOND_SWORD
  name: "&cInferno Blade"
  lore:
    - "&cInferno Set Weapon"
    - "&7Wield with the full set to set foes alight."
  trigger: ATTACK
  chance: 60
  effects:
    - { IGNITE: { duration: 60, who: "@Victim" } }
```

:::note Sets have no `tier`
Unlike enchants and crystals, an armor set carries no rarity tier. See
[Armor sets](../concepts/armor-sets.mdx).
:::

## Defining a crystal

A crystal lives in `content/crystals/`. It is a levelless trigger/effects block plus its own item
likeness (material/name/lore) so the minted crystal explains itself. From
`content/crystals/ember-crystal.yml`:

```yaml
tier: uncommon
display: "&6Ember Crystal"
description: "Ignite the enemy you strike."
material: BLAZE_POWDER         # the minted crystal item's look
name: "&6Ember Crystal"
lore:
  - "&7Socket into a sword or axe."
  - "&660%&7 chance on hit to set your"
  - "&7foe alight for &63 seconds&7."
trigger: ATTACK
applies-to: [SWORD, AXE]
chance: 60                     # no levels — one flat chance + effects
effects:
  - { IGNITE: { duration: 60 } }
  - { PARTICLE: { particle: FLAME, count: 8 } }
```

Two crystals can be **merged** into one paired socket. How many sockets an item has is set by
`crystals.slots` in `config.yml`. See [Crystals](../concepts/crystals.mdx).

## Customizing items

Every physical item is one file under `items/`, tagged by `type:`. These files describe each item's
**look** (material, name, lore) and its economy knobs — they never compile to an ability. See the
[Items & economy](../items/index.mdx) section for each item type in detail. A quick map:

| Item file | `type:` | Notable fields |
| --- | --- | --- |
| `enchant-book.yml` | `enchant-book` | `success-lore`, `destroy-on-fail`, `{ENCHANT}` / `{LEVEL}` |
| `white-scroll.yml` | `white-scroll` | protects gear from a failed enchant |
| `holy-white-scroll.yml` | `holy-white-scroll` | `save-chance` (keep items on death) |
| `black-scroll.yml` | `black-scroll` | `success-chance` (extract an enchant to a book) |
| `transmog-scroll.yml` | `transmog-scroll` | `name-suffix` (reorder enchant lore) |
| `godly-transmog.yml` | `godly-transmog` | reusable hand-pick reorder tool |
| `randomizer-scroll.yml` | `randomizer-scroll` | `min-percent`, `max-percent` |
| `unopened-book.yml` | `unopened-book` | `min-success`, `max-success`, `{TIER}` |
| `dust.yml` | `dust` | `min-bonus`, `max-bonus`, sound, particles |
| `soul-gem.yml` | `soul-gem` | `souls-per-kill`, `souls-per-mob`, colors, sounds |
| `crystal.yml` | `crystal` | `success-chance`, `consume-on-fail`, `extractor` |
| `slot-orb.yml` | `slot-orb` | `orb-amount`, `hard-cap` |
| `heroic.yml` | `heroic` | `success-min/max`, percents, `material-upgrades` |
| `nametag.yml` | `nametag` | `blacklist` (banned substrings) |

## Reloading

`/se reload` rebuilds the **entire** content library — `config.yml`, `lang.yml`, `content/`,
`items/`, and `menus/` — **off-thread**, then swaps it in atomically, and **only if it compiles
clean**. A broken edit keeps the running config live and reports diagnostics instead of taking the
server down.

```text
/se reload              # rebuild and hot-swap
/se reload --dry-run    # validate and report — do NOT swap
```

:::tip Validate before you commit
Run `/se reload --dry-run` after a big edit to catch typos in effect names, missing required params,
or bad references *before* you swap them into the live config.
:::

Use **config packs** (`/se pack export <name>`) to snapshot your whole setup into a portable ZIP,
and `/se pack apply <name>` to swap one in transactionally. See the
[commands](../commands/index.mdx) page.
