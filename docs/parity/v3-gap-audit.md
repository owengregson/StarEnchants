## Executive summary

The v3 directive is in strong overall shape: of the fourteen sections (A–N), four are fully delivered (B, H, L, M), and the bulk of the remaining content is substantially built, with gaps concentrated in two areas — the command surface (J) and the deliberately-last integrations wave (N). Across all confirmed gaps the count is **26 total: 7 High, 8 Medium, 11 Low**. The High-severity gaps cluster in the unified condition/variable/selector engine (A), crystal extraction (E), and the command layer (J), where multiple directive-mandated give-forms, targeting, and the inverse remove command are entirely absent. Section N's four gaps are all planned-not-started by design (see Notes).

## Confirmed gaps

### High

**Section A — No native compiler path emits Flow (%stop%/%force%/%allow%) or chance delta (±N %chance%)** (missing)
The directive requires making Flow/chanceDelta reachable via a compiler path emitting `%stop%`/`%force%`/`%allow%` and `±N %chance%` from condition clauses. The Expr AST has no clause/flow node; the parser recognises no `:` clause separator or sentinels; CompiledCondition holds only a Cond root; and ConditionEvaluator.eval returns only CONTINUE or STOP — never FORCE/ALLOW or a non-zero chanceDelta. Authored content cannot reach the existing Flow enum or gate-8 consumption through the native engine.
*Evidence:* `se/schema/src/schema/grammar/expr/Expr.java:33-97` (no flow node); `ExprParser.java:57-193` (no clause grammar); `se/compile/src/compile/model/CompiledCondition.java:21` (root-only); `se/engine/src/engine/condition/ConditionEvaluator.java:29-34` (only CONTINUE/STOP). The only sentinel-recognizing code is the AE migrator, which rejects `%force%`/`%chance%` as TODOs (`se/migrate/src/migrate/Mappings.java:513-519`).
*Recommended action:* Add a clause/flow/chance production to the Expr AST and parser, carry flow + chanceDelta fields through CompiledCondition, and emit them from ConditionEvaluator so gate 8 becomes reachable from authored content.

**Section A — Variable vocabulary far short of the Cosmic Enchants-style ~60 (static + dynamic + computed)** (partial)
BuiltinVars.vocabulary() declares only 22 facts (9 number, 9 flag, 4 string), and 2 of those (damage, combo) are declared-but-unsourced (read 0). Whole Cosmic Enchants-style categories are absent: actor entity-type, victim held-item, block-type/isBlock (needed for MINE), protected/white-scroll, weather/time/region, combat damage/combo, computed/derived variables, and parameterized dynamic vars beyond the SET_VAR passthrough.
*Evidence:* `se/engine/src/engine/condition/BuiltinVars.java:33-62` (22 declarations; damage/combo unsourced at :42-43); `se/engine/src/engine/run/FactPopulator.java:102-124` (no extractor for damage/combo/block/protected/entity-type-held/weather); `docs/parity/parity-audit.md:41-52,109-110`.
*Recommended action:* Port the full Cosmic Enchants-style vocabulary — add the missing static/dynamic/computed facts and wire FactPopulator extractors for each (this is the work that will trip MAX_FLAGS, so coordinate with the flag-widen below).

**Section A — Full Cosmic Enchants-style selector set not added — 11 named selectors absent (only 5 of the target set exist)** (missing)
Only Self, Victim, Attacker, Nearest, Aoe are registered. None of AllPlayers, PlayerFromName, EntityInSight, BlockInDistance, Block, EyeHeight, Add, Trench, Tunnel, Vein exist as selector kinds — they appear only in the migrator's "unmappable → TODO" branches. `@Nearest{filter=PLAYERS}` partially subsumes NearestPlayer/AllPlayers, but entity-in-sight, block-targeting, eye-height, location-offset, and mining-shape selectors have no equivalent.
*Evidence:* `se/engine/src/engine/selector/kind/BuiltinSelectors.java:20-26` (5 registered); selector kind directory holds only Self/Victim/Attacker/Nearest/Aoe + helpers; `se/migrate/src/migrate/Mappings.java:405,418` (named selectors → TODO).
*Recommended action:* Implement the 11 named selector kinds and register them in BuiltinSelectors, including the mining-shape selectors (Trench/Tunnel/Vein) and the block/eye-height/offset targeting selectors.

**Section E — Crystal extraction ("Extraction returns the multi-crystal as a whole") is entirely unimplemented** (missing)
There is no path to remove/extract a crystal from gear anywhere. The only operations on CombatState.crystals() are append-on-apply and carry-through-unchanged; the black scroll extracts enchants only. No extractor item, no `/se crystal-extract` command, and no removal gesture exist, so an applied crystal (or multi-crystal) can never be recovered.
*Evidence:* `se/feature/src/feature/crystal/CrystalService.java` (interact dispatches only merge/apply); `se/feature/src/feature/apply/ItemEnchanter.java:337` (append-only); `ScrollService.java:228` (crystals carried unchanged); `docs/parity/parity-audit.md:281,290-291`.
*Recommended action:* Add a crystal-extract drag gesture (per §K) and/or command that removes a crystal-list entry from gear and returns the multi-crystal as a single whole item.

**Section J — No `<player>` target on any give-command — all give to the sender only** (missing)
Every directive give-form takes an explicit `<player>` target, but all implemented give verbs require the sender to be a player and give to that sender; none parse or resolve a target. There is no `Bukkit.getPlayer`/`getPlayerExact` call in the command or feature give paths.
*Evidence:* `se/bootstrap/src/bootstrap/SeCommand.java` give methods (191-203, 402-423, 426-437, 481-541) all start `if (!(sender instanceof Player player))` then add to the sender's inventory; `docs/parity/parity-audit.md:197,200,206,222`.
*Recommended action:* Add `<player>` target resolution (online-player lookup) to all give-commands, defaulting to the sender when omitted, and surface a missing/offline-player diagnostic.

**Section J — `give item <player> <item-id> [args]` universal dispatcher is absent** (missing)
The directive requires a single `give item` dispatcher routing to all item types by id (soul-gem `<amount>`, scrolls, orb/slot-gem, nametag, dust, crystal `<id>`). No such dispatcher exists; item types are reachable only via individual flat verbs, several with no args (e.g. `gem` mints a 0-soul gem with no `<amount>`), and dust is unreachable entirely.
*Evidence:* `SeCommand.java` SUBCOMMANDS (48-51) and switch (104-130) have no `item` route; `gem` (191-203) takes no amount; `docs/parity/parity-audit.md:200,206`.
*Recommended action:* Add a `give item <player> <item-id> [args]` parent dispatcher routing by item id to each item's mint path, including per-item arg parsing (soul amount, crystal id, etc.).

**Section J — `give set <player> <set> <piece|weapon>` and `give upgrade` are missing** (missing)
The directive lists `/se give set` and `/se give upgrade` as distinct verbs. There is no set-piece give command and no set-minting service (content/sets/ exists but no SetService/mintSet code), and no `upgrade` verb distinct from `heroic`.
*Evidence:* No `*set*service*`/`mintSet`/`setPiece` in se/**; `SeCommand.java:48-51` has neither `set` nor `upgrade`; `SetsBrowserMenu.java:13-20` is read-only and notes no member-piece list exists; `docs/parity/parity-audit.md:222-223`.
*Recommended action:* Build a set-piece minting model + service and add `/se give set <player> <set> <piece|weapon>`; add the distinct `/se give upgrade` verb.

**Section J — Inverse `/se removeenchant` / `unenchant` is entirely absent** (missing)
The directive explicitly requires the inverse remove command. No subcommand, alias, or backing API exists; ItemEnchanter exposes only check*/apply* methods with no remove/unapply/strip method.
*Evidence:* grep for `removeenchant|unenchant|removeEnchant` in se/** returns nothing; `SeCommand.java:48-51,104-130` omit it; `ItemEnchanter.java` has no remover; `docs/parity/parity-audit.md:195,240`.
*Recommended action:* Add a `removeEnchant` backing API to ItemEnchanter (recomputing slots/lore from state) and wire `/se removeenchant` / `unenchant` with `<player>`/enchant args.

### Medium

**Section C — MODIFY_MONEY missing the STEAL_MONEY_PERCENT (percent transfer) variant** (partial)
The §C merge table lists MODIFY_MONEY as replacing STEAL_MONEY[_PERCENT], where `[_PERCENT]` is a percent-of-balance steal. MoneyEffect implements only a fixed `amount` with give/take/transfer modes; there is no percent mode and EconomyService exposes no balance accessor, so the percent transfer cannot be expressed. Fixed STEAL_MONEY is covered via mode=transfer; only the percent variant is absent.
*Evidence:* `se/engine/src/engine/effect/kind/MoneyEffect.java:30-38` (amount-only, no percent mode); `se/platform/src/platform/economy/EconomyService.java` (no balance() accessor); `docs/v3-directives.md:99`; `docs/parity/parity-audit.md:90,333,420`.
*Recommended action:* Add a percent mode to MODIFY_MONEY and a balance-read on the economy seam so a fraction-of-balance transfer can be computed at the effect layer.

**Section D — Soul-gem particles never spawned (while-active / on-activate / on-deactivate all unimplemented)** (partial)
Directive D requires multiple particles-while-active / -on-activate / -on-deactivate. The fields are parsed into SoulGemConfig and exposed in YAML, but no code ever spawns them — there is no spawnParticle call in the soul feature, toggle() emits no particles, and there is no while-active loop. *Recommended action:* Wire the three particle knobs into the soul-gem lifecycle — spawn on-activate/on-deactivate from toggle(), and emit while-active particles via the HELD/REPEATING equip loop.

**Section E — Configurable apply/remove SOUNDS for the crystal gesture are missing** (missing)
The directive requires "apply/remove messages & sounds (all standard, configurable)." The sibling soul-gem feature already implements the exact configurable-sound pattern, confirming this is a shortfall rather than an unavoidable deferral.

**Section F — reduction-scope=ALL (all-causes) is parsed but inert — only ENTITY/PvP is ever applied** (partial)
The directive requires heroic reduction default to entity/PvP but be configurable to all-causes. HeroicConfig.reductionScope is parsed and defaulted to ENTITY, but no runtime code reads reductionScope(); heroic reduction is contributed only via the EntityDamageByEntityEvent path, so setting reduction-scope: ALL has zero effect. The ENTITY default behaves correctly; the gap is the configurability, and the ALL option is offered to users while being non-functional.
*Evidence:* grep for reductionScope() returns only the declaration + a normalization test, no consumer; `se/feature/src/feature/trigger/TriggerRunner.java:97` reached only via `CombatDispatch.java:172-178`; `docs/decisions/0021-heroic-multiplicative-stage.md:52-54` (deferred); `items/heroic.yml:18-19` (ALL offered).
*Recommended action:* Gate heroic reduction on reductionScope — when ALL, apply reduction in the environmental EntityDamageEvent (non-by-entity) path as well; otherwise restrict to entity/PvP.

### Low

**Section A — FactBuffer flag representation not widened (deferred, but trigger condition not yet met)** (deviation)
FactBuffer still uses a single 64-bit long with MAX_FLAGS=Long.SIZE; there is no second word, boolean-to-number promotion, or overflow handling. This is a justifiable deferral — the widening was conditioned on the boolean vocab approaching 64, and only 9 flags are currently declared — but it is coupled to the unfilled "~60 variables" work, since porting the full Cosmic Enchants-style boolean vocab is what would trip MAX_FLAGS.
*Evidence:* `se/engine/src/engine/condition/FactBuffer.java:20-21,25,32-34` (single long, hard reject >64); `VarVocabulary.java:116-118` (build() throws if flagSlots>64); only 9 flags in `BuiltinVars.java:45-55`; no flags2/secondWord/widen/promote in se/engine or se/compile.
*Recommended action:* Address jointly with the §A vocabulary port — widen the flag representation (second word or boolean-to-number promotion) at the point the boolean vocab approaches 64.
