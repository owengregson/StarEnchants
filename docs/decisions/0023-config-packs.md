# ADR 0023: Config packs (swappable whole-config presets)

- **Status:** Accepted
- **Date:** 2026-06-23
- **Deciders:** project owner + agent
- **Relates to:** ADR 0014 (content loader + transactional reload), ADR 0006 (config & migration),
  ADR 0016 (content-format v2); the first shipped pack is the EliteEnchantments port

## Context

An operator wants to swap their entire StarEnchants configuration to a curated preset in one action —
"give me the EliteEnchantments setup" — and swap back, without hand-editing dozens of files. The plugin
already extracts a default config surface on first boot (`config.yml`, `lang.yml`, `content/`, `items/`,
`menus/`) and reloads all of it atomically via the `ContentReloader` (every source swapped in one
all-or-nothing transaction). A pack is the natural unit on top of that: a named, portable snapshot of
the whole surface that can be exported, shipped, and applied.

Two product decisions were the operator's to make:

1. **On-disk format.** A single shareable file vs. a folder. Chosen: a **ZIP archive** (`packs/<name>.zip`)
   — one artifact to hand someone, standard, compact. The repo keeps the shipped pack as a *reviewable*
   config tree (`se/bootstrap/packs-src/<name>/`) zipped into the jar at build time, so PRs diff the YAML
   while the shipped/exported artifact is the chosen ZIP.
2. **Apply safety.** Chosen: **auto-backup first** — applying a pack snapshots the current config into a
   timestamped backup pack before overwriting, so a swap is always reversible (`/se pack apply <backup>`).

## Decision

A new dependency-free module `se/pack` owns the format and the on-disk store:

- **`PackArchive`** — the pure codec: a tree of config files ↔ a ZIP, with a `pack.yml` manifest entry
  (name/description/author/format/created/fileCount). Deterministic (sorted entries, zeroed timestamps),
  so a given surface yields a byte-identical archive. `peekManifest` reads only the header for listings.
- **`PackSurface`** — defines the surface (`config.yml`, `lang.yml`, `content/`, `items/`, `menus/`) and
  the collect / write / clear / promote primitives. Apply is **stage-then-swap**: extract the new surface
  into a staging dir (fails before touching anything live), back up the current surface, clear the live
  surface, then move staging into place — so a failed write never half-clobbers the config. Entries that
  escape the surface (a foreign top-level name, a `..`) are skipped, so an untrusted pack can only ever
  write inside the five surface roots.
- **`PackStore`** — `list` / `info` / `export` / `apply` over the `packs/` directory.

The composition root wires `PackStore.apply` to the existing transactional reloader: the on-disk swap is
followed by the same atomic reload as `/se reload`, so a swapped pack takes effect live (and a pack with a
*content* fault keeps the previous in-memory state and reports the diagnostics). `/se pack <list|info|apply
|export>` is the surface; shipped packs extract on first boot via `packs/index.txt`, exactly like
`content/`.

The first shipped pack — **`elite-enchantments`** — is the full EliteEnchantments port: 122 enchants
(generated by the extended EE migrator, with EE rarity groups carried as both the SE `tier:` and the SE
`group:`, `REPEATING-<n>` mapped to the REPEATING trigger + period, and the EE effect/condition vocabulary
translated faithfully — seconds→ticks, random ranges→max, `DISABLE_ENCHANTMENT*`→`SUPPRESS`,
`isPlayerHealth`/`isTargetHealth`/`isPlayerBlocking`→facts), plus the standard items/config/lang/menus
surface. A build-time validity test compiles the whole pack clean before it can ship.

## Consequences

- An operator swaps the entire config with one command, reversibly. Sharing a config is one file.
- "Add a content preset" is local: drop a reviewable tree under `packs-src/<name>/`, register a `Zip`
  task, list its archive in `packs/index.txt`. The codec and command are untouched.
- The EE migrator is now a faithful, tested EE-vocabulary translator (not just the verified core), reusable
  by `/se migrate ee`. Genuinely exotic EE mechanics with no SE equivalent (ROT_DECAY, SNIPER, GANK,
  AUTO_LOCK, …) remain honest `# TODO` lines — the enchant still loads without that one effect.
- **Boot-gated wiring** (the souls/slots/scrolls listeners, integration discovery, the command-trigger)
  binds once at enable, so a pack that flips those toggles needs a server restart to fully take effect; the
  apply feedback says so. Live combat/feature values re-read on the reload.
- The matrix tester loads only the engine via the tester jar, so the pack command, first-boot extraction,
  and the EE pack's per-version handle resolution are not exercised by the live matrix today. The pack
  system + migrator are pure logic (unit-tested); the EE pack content is structurally validated at build
  time; per-version handle tokens degrade gracefully (resolver warn-skip). A future `ElitePackSuite`
  (bundling the pack into the tester jar) could live-validate the EE handles per version if wanted.

## Alternatives considered

- **A readable text bundle** (length-framed concatenation) instead of a ZIP — more diffable, but the repo
  already keeps a reviewable source tree, so the on-disk artifact's readability mattered less than being a
  standard single file; ZIP won.
- **A pack folder** (`packs/<name>/`) instead of a single file — simplest (no codec), but not a single
  shareable artifact, which is the whole point of "hand someone a pack."
- **Straight overwrite on apply** (no backup) — simpler, but a destructive swap with no way back; rejected
  for the auto-backup.
- **Hand-porting the 122 EE enchants** instead of extending the migrator — brittle, inconsistent, and
  throws away reusable migration coverage; rejected for the migrator extension + generator.
